## ŌÜĪ What it does

- **Scaffold** a problem folder from a Kattis / Codeforces / LeetCode URL. Sample test cases are auto-fetched.
- **Test** your `Solution.java` against every `*.in` / `*.out` pair with a single click.
- **Commit** with a Conventional-Commits message generated by Claude from your staged diff.
- **Browse** every solved problem in a sidebar grouped by platform. Newly added problems are auto-focused in the tree and their solution opened in the editor.
- **Playground** ŌĆö a dedicated `playground/Playground.java` for ad-hoc Java experimentation. One click compiles and runs it.

## ­¤Ä» Daily workflow

1. **Open VS Code on a folder** ŌĆö either an existing CP folder or a brand-new one.
2. **First time only** (required): run **KestrelCP: Initialize Workspace** from the command palette to create the `kattis/`, `codeforces/`, `leetcode/`, and `playground/Playground.java` scaffolding. All other commands require the workspace to be initialized first.
3. **Click the cube icon** in the activity bar ŌåÆ the **Problems** sidebar lists every problem grouped by platform.
4. **Scaffold a new problem**:
   - `+` button at the top of the Problems sidebar, or
   - command palette ŌåÆ **KestrelCP: New Problem**

   Pick platform, paste the problem URL, hit Enter. A `KestrelCP` terminal shows the scraper running; sample tests are auto-fetched into `<platform>/<problem>/`. The new problem is automatically focused in the sidebar and `Solution.java` opens in the editor.

5. **Solve it.**
6. **Run tests** ŌĆö two ways:
   - Hover the problem in the sidebar ŌåÆ click the inline Ō¢Č button
   - Command palette ŌåÆ **KestrelCP: Run Tests for Current File**
7. **Commit** ŌĆö stage your files in the Source Control panel, then command palette ŌåÆ **KestrelCP: AI Commit**. Claude proposes a Conventional Commit message; type `y` / `n` / `e` to accept / abort / edit.

### ­¤¦¬ Playground

Need to try a snippet without scaffolding a whole problem? **KestrelCP: Initialize Workspace** also creates `playground/Playground.java` ŌĆö a free-form Java scratchpad. Open it from the **Playground** entry at the top of the sidebar (beaker icon) and hit the inline Ō¢Č button to compile + run. Add helper classes alongside it (`Helper.java`, etc.) ŌĆö `javac *.java` picks them up automatically; `java Playground` is the entry point.

### ­¤öä Repairing existing problems

When a scraper is fixed in a new release, or a site (Kattis / Codeforces / LeetCode) changes its HTML/API, your pre-existing `*.in` / `*.out` files won't auto-update ŌĆö they're already on disk. To re-pull them across every problem in your workspace:

- **Cmd+Shift+P** ŌåÆ **KestrelCP: Refetch All Sample Tests**

A confirmation modal shows the per-platform problem counts (e.g. _kattis: 2, codeforces: 5, leetcode: 1_). On confirm, every problem is re-scraped using the URL stored in its `notes.md`. Only `*.in` and `*.out` are overwritten ŌĆö `Solution.java` and `notes.md` are untouched.

| Command                                 | What it does                                                                                                                                                                                      |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `KestrelCP: Initialize Workspace`       | Creates platform directories and `playground/Playground.java`. Required before using any other command.                                                                                           |

## ­¤öŹ How it works

- **Scaffolding** (Kattis / Codeforces) parses HTML with `requests` + `BeautifulSoup` and writes one `*.in` / `*.out` pair per sample. **LeetCode** uses the public GraphQL endpoint; each `.in` file holds one parameter per line, so `Solution.main` reads arguments with `sc.nextLine()`. Premium-only and design-style problems (LRU Cache, etc.) don't fit the stdin/stdout model.
- **Testing** runs `javac Solution.java`, then `java Solution` once per `*.in` with stdin redirected. Output is compared to the matching `*.out` ignoring trailing whitespace and blank lines. 5-second timeout per case.
- **AI Commit** runs `git diff --staged`, asks Claude for a one-line Conventional Commit message with an emoji prefix:

| Emoji | Type       | When to use                       |
| ----- | ---------- | --------------------------------- |
| Ō£©    | `feat`     | New solution or feature           |
| ­¤Éø    | `fix`      | Bug fix                           |
| ­¤ōØ    | `docs`     | Notes / README update             |
| ŌÖ╗’ĖÅ    | `refactor` | Rewrite without changing behavior |
| Ō£ģ    | `test`     | Adding or fixing test cases       |
| ­¤ö¦    | `chore`    | Tooling, config, scripts          |
| ŌÜĪ’ĖÅ    | `perf`     | Performance improvement           |
| ­¤Æä    | `style`    | Formatting only                   |

Confirm with `y` (accept), `n` (abort), or `e` (edit before committing).
